# GodBlade 뽑기(Gacha) 시스템 기획서 — NewGachaPattern

> **작성일**: 2026-03-06
> **프로젝트**: GodBlade (Evil of Darkness)
> **SIGIL Stage**: S3 Design Document
> **기반 문서**: 이해도 리포트 v1.1, 에이전트 회의 결과 (게임 디자이너 / 서버 아키텍트 / 수익화 전문가)
> **프로토콜 전략**: BuyGacha 100% 재사용 (Request/Response 변경 없음)

---

## 1. 시스템 개요

### 1.1 목적

NewGachaPattern 테이블 기반의 새로운 뽑기 시스템을 구현한다. 기존 확률 기반 뽑기(ShopGachaGradeGroup → ShopGachaItemGroup)와 달리, **사전 정의된 패턴(Pattern01~PatternN)의 당첨금 값을 순차 소비**하는 구조이다. 슬롯 수(N)는 Row별 `PatternCount` 필드로 정의되며 고정값이 아니다.

### 1.2 핵심 차이점 (기존 vs NewGachaPattern)

| 항목 | 기존 뽑기 (ShopGacha) | NewGachaPattern |
|------|----------------------|-----------------|
| 당첨 결정 | 확률 기반 (GradeGroup → ItemGroup) | 패턴값 기반 (Pattern01~N 사전 배치, N은 가변) |
| 보상 구조 | 아이템 직접 지정 | 당첨금 → 분해 → 쌍 아이템 |
| UI 표시 | 뽑힌 아이템만 | **항상 10개** (보상 + 필러) |
| 유저 추적 | ShopBuyCount | 복합키 (GachaPriceCode_PageGroup_ItemGroupCode) |
| Row 관리 | 없음 | 유저별 Row 할당/해제/갱신 |
| 천장(Pity) | 없음 | N슬롯 = 사실상 천장 + 소프트 피티(연속 꽝 10회 보장) |

### 1.3 프로토콜 전략 — BuyGacha 재사용

기존 BuyGacha 프로토콜을 100% 재사용한다. NewGachaPattern용 상품을 ShopGacha 테이블에 등록하고, 서버의 `BuyGachaService`에서 `gachaType` 분기로 처리한다.

| 레이어 | 변경 여부 | 설명 |
|--------|:---------:|------|
| BuyGacha Request | 없음 | `productCode`, `onceMore`, `isFreeRequest` 그대로 |
| BuyGacha Response | 없음 | 기존 필드 매핑으로 충분 |
| BuyGachaEvent (클라이언트) | 없음 | 이벤트 구조 동일 |
| 클라이언트 UI/연출 | **변경** | 결과 팝업 10개 표시 + 보상/표시 구분 연출 |
| BuyGachaService (서버) | **분기 추가** | `gachaType == NewPattern` → 전용 로직 |
| 서버 데이터 | **추가** | NewGachaPattern, NewGachaItemGroup 테이블 + 할당 테이블 |

## 2. 패턴 Row 할당 플로우

### 2.1 전체 흐름

```
유저가 뽑기 시도
    |
    v
GachaPriceCode 매칭 (100 / 1000 / 10000)
    |
    v
해당 price_code에 기존 할당 row가 있는가?
    |
    +-- YES --> 해당 row의 다음 Pattern 슬롯 소비 (순차)
    |
    +-- NO  --> 새 row 할당 프로세스
                    |
                    v
                GachaPageGroup 오름차순 체크
                (PageGroup N의 미할당 row 전부 소진 시 N+1로 이동)
                    |
                    v
                같은 PageGroup 내 미할당 row 중 랜덤 1개 선택
                    |
                    v
                DB 원자적 선점 (Optimistic Claim)
                    |
                    v
                복합키 부여 (GachaPriceCode_GachaPageGroup_GachaItemGroupCode)
                    |
                    v
                Pattern01 소비 (첫 슬롯)
```

### 2.2 복합키 규칙

- 유저는 price_code별 **1개씩**, 최대 **3개** 복합키 보유 (100, 1000, 10000)
- **복합키 중복 지급 금지** — 같은 row가 여러 유저에게 할당되지 않음 (DB 원자적 선점으로 보장)
- Pattern01~N **전부 소진** 시 (N = Row의 `PatternCount`):
  - 기존 복합키 해제 (ClaimedByPlayerId = NULL로 리셋)
  - 다음 뽑기 시 새 row 할당 (위 프로세스 반복)
  - 새 복합키 부여

### 2.3 Row 할당 동시성 제어 — DB 선점 방식

NewGachaPatternAssignment 테이블로 관리:

| 컬럼 | 타입 | 설명 |
|------|------|------|
| GachaPriceCode | int | PK 일부 |
| GachaPageGroup | int | PK 일부 |
| GachaItemGroupCode | int | PK 일부 |
| ClaimedByPlayerId | bigint (nullable) | NULL = 미할당 |
| CurrentSlotIndex | int | 0 ~ (PatternCount-1) (순차 소비 위치) |
| ClaimedAt | datetime | 할당 시각 |

할당 시 `UPDATE ... WHERE ClaimedByPlayerId IS NULL LIMIT 1` 원자적 쿼리로 동시 요청 충돌을 방지한다.

### 2.4 서브그룹(GachaItemGroupCode) 역할

**같은 가격대(GachaPriceCode)의 서로 다른 패턴 변형**이다. 유저마다 다른 row를 할당받으므로 결과 다양성을 확보한다. 이벤트/시즌별로 새 패턴 테이블을 추가할 때 기존 유저 배정에 영향 없이 확장 가능하다.

### 2.5 패턴 슬롯 소비 — 순차 (01→02→...→N)

- 1회 뽑기 = `CurrentSlotIndex` 위치의 Pattern값 소비 후 인덱스 +1
- `CurrentSlotIndex` int 1개로 상태 관리 (DB I/O 최소화)
- 서버 전용 데이터이므로 클라이언트가 패턴 순서를 알 수 없음 (역공학 리스크 없음)
- 순차 소비로 기획자가 **경험 커브 설계** 가능 (초반 소액 당첨 집중, 후반 대형 당첨 배치 등)

## 3. 뽑기 실행 플로우

### 3.1 꽝 처리 (패턴값 = 0)

```
패턴값 == 0
    |
    v
소프트 피티 체크: 연속 꽝 10회 도달?
    |
    +-- YES --> 미사용 슬롯 중 양수값 1개를 강제 소비 (당첨 보장)
    |
    +-- NO  --> 꽝 처리 진행
                    |
                    v
                실제 보상 없음 (인벤토리 미추가)
                    |
                    v
                필러 10개 선정 (고등급 1~2개 포함)
                    - 고등급(ItemCode→Item.GeneralTypeCode 조인→ItemGrade 7~11) 1~2개
                    - 등급별 확률(7성60%/8성20%/9성10%/10성5%/11성5%)
                    - 나머지: RequiredClass 매칭 랜덤
                    - ItemCode 중복 없이 각 1개
                    - 전부 표시용 (인벤토리 미추가)
```

**Response 매핑 (꽝)**:
- `equipmentItems` = null
- `worthes` = null
- `gachaResult` = Worth[10] (표시용 10개)

### 3.2 당첨 처리 (패턴값 > 0)

```
패턴값 > 0 (= 당첨금 총액)
    |
    v
당첨금을 GachaItemPriceCode 단위로 분해
    |
    v
실시간 재귀 열거 (Safety Cap: 200개)
    |
    v
가격대별 가중치로 1개 조합 선택
    |
    v
각 분해 단위 = 1쌍(pair) = 같은 아이템 2개
    - GachaItemPriceCode + RequiredClass 매칭 풀에서 랜덤 1개 선택 x2
    - 쌍 간 서로 다른 ItemCode (중복 금지)
    - 실제 보상 (인벤토리 추가)
    |
    v
남은 슬롯 = 10 - (분해 단위 수 x 2)
    |
    v
남은 슬롯만큼 필러 아이템 선정 (고등급 1~2개 포함)
    - 고등급(ItemCode→Item.GeneralTypeCode 조인→ItemGrade 7~11) 1~2개
    - 등급별 확률(7성60%/8성20%/9성10%/10성5%/11성5%)
    - 나머지: RequiredClass 매칭, 가격대 무관
    - 이미 선택된 쌍 아이템 ItemCode 제외
    - ItemCode 중복 없이 각 1개
    - 표시용 (인벤토리 미추가)
    |
    v
총 10개 = 당첨 쌍 아이템(보상) + 랜덤 필러(표시용)
```

**Response 매핑 (당첨)**:
- `equipmentItems` = 쌍 아이템 (장비인 경우, 실제 보상)
- `worthes` = 쌍 아이템 (재화인 경우, 실제 보상)
- `gachaResult` = Worth[10] (보상 + 필러 전체, UI 표시용)
- `bonusEquipmentItems` = null
- `bonusWorthes` = null

### 3.3 소프트 피티 메커니즘

| 항목 | 값 |
|------|-----|
| 트리거 | 연속 꽝 10회 도달 시 |
| 동작 | 다음 Pattern 소비를 양수값 슬롯으로 강제 교체 |
| 리셋 | 당첨 시 연속 꽝 카운터 = 0 |
| 저장 | 유저별 DB (ConsecutiveLossCount) |

### 3.4 전체 1회 뽑기 플로우 (종합)

```
[유저 뽑기 시도]
        |
        v
[1] Row 할당 확인/할당 (섹션 2)
        |
        v
[2] CurrentSlotIndex 위치의 Pattern값 소비
        |
        +--- 패턴값 == 0 + 소프트피티 미도달 ---> [3a] 꽝 처리
        |
        +--- 패턴값 == 0 + 소프트피티 도달 -----> [3c] 피티 발동 → 당첨 처리
        |
        +--- 패턴값 > 0 -----------------------> [3b] 당첨 처리
        |
        v
[4] CurrentSlotIndex += 1
[5] Pattern 사용 히스토리 갱신 (연속 꽝 카운터 업데이트)
        |
        v
[6] 전체 슬롯(PatternCount) 소진? → YES → 복합키 해제, 다음 뽑기 시 새 row 할당
        |
        v
[7] "새로운 패턴 시작" 연출 (Row 소진 시, 섹션 8.3 참조)
```

## 4. 당첨금 분해 알고리즘

### 4.1 유효 분해 단위 (GachaItemPriceCode)

```
500, 1000, 2500, 5000, 10000, 25000, 50000, 100000, 250000, 500000, 1000000, 2500000
```

### 4.2 분해 규칙

1. 당첨금 총액을 위 유효 단위들의 조합으로 **정확히** 분해
2. 각 단위 = 2아이템(1쌍) → **최대 5단위** (10아이템 상한)
3. 분해 단위 합 == 당첨금 총액 (과부족 불가)
4. 당첨금 이하의 GachaItemPriceCode만 사용 (수학적 필연)

### 4.3 알고리즘 — 실시간 재귀 열거 + Safety Cap

**Bounded Coin Change** 문제로 모델링:

```
DecomposeAmount(remaining, maxUnits=5, minUnitIndex, currentCombo, results, cap=200)
  if remaining == 0: results.Add(currentCombo.Copy()); return
  if maxUnits == 0 || results.Count >= cap: return
  for i = minUnitIndex to units.Length-1:
    if units[i] > remaining: continue
    currentCombo.Add(units[i])
    DecomposeAmount(remaining - units[i], maxUnits - 1, i, ...)
    currentCombo.RemoveLast()
```

- `minUnitIndex`로 중복 조합 방지 (정렬된 조합만 생성)
- `cap=200`으로 성능 보호 (실제로 도달하는 경우는 드물 것으로 예상)
- **성능**: 깊이 5 x 가지치기 → 대부분의 패턴값에서 **1ms 이내** 완료

### 4.4 조합 선택 — 가격대별 가중치

균등 랜덤 대신 **가중치 선택**을 적용한다. 조합 내 최고가 단위를 기준으로 가중치를 산정한다.

| GachaPriceCode | 가중치 방향 | 근거 |
|:--------------:|:----------:|------|
| 100 | 다단위 분해 우선 (풍성함) | 저가 유저 만족도 극대화 → 재과금 유도 |
| 1000 | 밸런스 (균등 가중치) | 다양한 경험 제공 |
| 10000 | 단일/소수 분해 우선 (희소성) | 고가 유저에게 프리미엄 체감 |

**대형 당첨금(100,000원 이상)**: 최고 단가 단위를 1개 이상 포함하는 조합에 60~70% 가중치. "대형 당첨 = 좋은 아이템"이라는 유저 직관을 지지한다.

### 4.6 데이터 무결성 — 서버 기동 시 전수 검증

서버 기동 시 NewGachaPattern의 모든 Pattern01~PatternN 값에 대해 (N = 해당 Row의 `PatternCount`):
- 0 (꽝) → OK
- 양수 → 분해 가능 여부 검증. 불가 시 ERROR 로그 + 해당 슬롯 0 처리 (안전 폴백)

이로써 런타임 분해 실패가 원천 차단된다.

## 5. 아이템 선정

### 5.1 쌍(Pair) 아이템 선정

```
분해 단위 1개 (예: GachaItemPriceCode = 1000)
    |
    v
NewGachaItemGroup에서 GachaItemPriceCode == 1000 필터링
    |
    v
RequiredClass == character_class_type 필터링
    |
    v
이전 쌍에서 이미 선택된 ItemCode 제외 (쌍 간 중복 금지)
    |
    v
후보 풀에서 랜덤 1개 선택
    |
    v
같은 아이템 x 2 = 1쌍 (pair)
    → 인벤토리에 2개 추가 (실제 보상)
```

- **쌍 간 중복 금지**: 같은 세션에서 각 쌍은 서로 다른 ItemCode
- **쌍 내부**: 같은 아이템 2개
- **Fallback**: 아이템 풀 부족 시 중복 허용 + 운영 로그 기록

### 5.2 필러 아이템 선정 — 고등급 의도 배치

필러는 단순 랜덤이 아니라, **고등급 아이템 1~2개를 의도적으로 포함**하여 "다음엔 이걸 뽑을 수 있겠다"는 기대심리를 유발한다.

```
남은 슬롯 수 = 10 - (분해 단위 수 x 2)
    |
    v
[Step 1] 고등급 필러 선정 (1~2개)
    - NewGachaItemGroup.ItemCode == Item.GeneralTypeCode 조인 → Item.ItemGrade 참조
    - ItemGrade 7~11 아이템 풀에서 등급별 확률로 선택
    - 등급별 확률: 7성 60%, 8성 20%, 9성 10%, 10성 5%, 11성 5%
    - RequiredClass 매칭 + 이미 선택된 쌍 아이템 ItemCode 제외
    |
    v
[Step 2] 일반 필러 선정 (나머지)
    - RequiredClass 매칭 (가격대 무관)
    - 이미 선택된 아이템(쌍 + 고등급 필러) ItemCode 제외
    - 남은 풀에서 중복 없이 랜덤 선택
    |
    v
전체 필러 = 고등급 필러 + 일반 필러
    → 표시용만 (인벤토리 미추가)
```

**고등급 필러 등급별 확률**:

| 등급 | 확률 | 의도 |
|:----:|:----:|------|
| 7성 | 60% | 자주 보여서 "곧 뽑힐 것 같은" 기대감 |
| 8성 | 20% | 중간 희귀도로 관심 유발 |
| 9성 | 10% | 간헐적 노출로 욕구 자극 |
| 10성 | 5% | 드물게 보여서 강한 인상 |
| 11성 | 5% | 최고 등급 노출로 과금 동기 극대화 |

> **핵심 의도**: 필러는 "내 것이 아닌" 아이템이지만, 고등급이 섞여 있으면 "다음 뽑기에서 저걸 받을 수 있다"는 기대심리가 형성되어 재과금을 유도한다.

### 5.3 아이템 풀 사전 검증 (서버 기동 시)

- RequiredClass별 전체 아이템 수 >= 10 (꽝 시 10개 필요)
- GachaItemPriceCode별 + RequiredClass별 아이템 수 >= 5 (최대 5단위 분해 시 5개 필요)
- 부족 시 WARNING 로그

### 8.2 상자 개봉 연출

뽑기 실행 시 보물상자 등장→개봉→카드 배치의 3단계 연출을 수행한다.

```
[Step 1] 보물상자 등장
    - 화면 중앙에 보물상자 등장 (바운스/진동 애니메이션)
    - 가격대별 상자 외형 차별화 (100: 나무상자 / 1000: 은상자 / 10000: 금상자)

[Step 2] 상자 개봉
    - 탭 또는 자동으로 상자 뚜껑 열림
    - 빛줄기 이펙트 + 개봉 사운드
    - 가격대에 따른 빛 색상 (8.5 참조)

[Step 3] 카드 10장 배치
    - 상자에서 카드 10장이 날아와 2열 x 5장으로 배치
    - 모든 카드는 뒷면(뒤집힌) 상태
    - 배치 완료 후 조작 가능 상태로 전환
```

#### 상자 에셋 Fallback (SB-3)
- **1순위**: 직접 제작 (줌인+광원+플래시 파티클 시퀀스)
- **2순위 (Fallback)**: Unity Asset Store — `Treasure Chest with Effects`
  - URL: https://assetstore.unity.com/packages/3d/props/treasure-chest-with-effects-263013
  - 조건: 1순위 직접 제작이 어려운 경우에만 사용
  - 라이선스: Asset Store EULA 확인 필요

### 8.3 카드 조작 — 수동/자동 양방향

유저에게 **수동 선택**과 **자동 전체 공개** 두 가지 조작 방식을 제공한다.

#### 수동 조작

- 카드를 **개별 탭**하여 1장씩 뒤집기
- 뒤집힌 카드에 아이템 정보 + 가격대 이펙트 표시
- 10장 전부 뒤집으면 짝 맞추기 연출(8.4)로 전환

#### 카드 뒤집기 상세 타이밍 (DOTween Sequence)

```
총 0.6초:
- 0~0.3초: Y축 0° → 90° 회전 (앞면 숨김)
  - 중간점(90°): UISprite 교체 (뒷면 → 앞면 스프라이트)
- 0.3~0.6초: 90° → 180° 회전 (앞면 공개)
  - 완료 시: 등급 이펙트 트리거
```

DOTween Sequence 구현 예시:
```csharp
Sequence seq = DOTween.Sequence();
seq.Append(cardObj.transform.DORotate(new Vector3(0, 90, 0), 0.3f));
seq.AppendCallback(() => { sprite.spriteName = frontSpriteName; });  // NGUI UISprite 교체
seq.Append(cardObj.transform.DORotate(new Vector3(0, 180, 0), 0.3f));
seq.AppendCallback(() => { TriggerGradeEffect(); });
```

#### 자동 조작 ("모두 열기")

카드를 **랜덤으로 하나씩** 자동 오픈하며, 짝이 발견되면 그 자리에서 즉시 연출 후 계속 진행한다.

#### 자동 조작("모두 열기") 오픈 순서 (SB-4 확정)
- 카드 10장 중 오픈 순서는 **랜덤으로 하나씩** (순차가 아님)
- 구현: Fisher-Yates 셔플 or Random.Range로 미오픈 카드 인덱스 랜덤 선택
- Q4 확정: 하나씩 열면서 짝 발견 즉시 연출 → 연출 후 계속 진행 반복

```
루프 (남은 카드 있는 동안):
  1. 미오픈 카드 중 랜덤 1장 선택 오픈 (뒤집기 애니메이션)
  2. 짝 확인 — 이미 열린 카드 중 동일 아이템 존재하는가?
     └─ Yes → 짝 맞추기 연출(8.4) 즉시 실행 → 연출 완료 후 루프 계속
     └─ No  → 다음 카드로 바로 진행
  3. 반복
```

- 카드 간 딜레이: 0.2~0.3초 (짝 연출 시간 제외)
- 짝 연출 완료 후 자동으로 다음 카드 진행 (유저 입력 불필요)
- 10장 전부 공개 완료 → 결과 화면(8.9)으로 전환

> **Q4 확정**: "모두 열기" = 하나씩 열면서 짝 발견 즉시 연출 후 계속 진행 (2026-03-16)

### 8.4 짝 맞추기 연출 — 3단계

#### 짝 맞추기 트리거 시점 (SB-5 + Q4 확정)
- **트리거 조건**: 카드를 뒤집는 과정 중 이미 뒤집어진 카드 중 동일 아이템 발견 즉시 연출
- **적용 범위**: 수동(탭) + 자동("모두 열기") 모두 동일하게 적용
- **전체 공개 후 일괄 아님** — 개별 카드 뒤집기 완료 시점마다 즉시 판정
- **연출 후 계속**: 짝 맞추기 연출 완료 후 다음 카드 오픈 계속 진행

10장 전부 공개 후, 쌍(pair) 아이템을 시각적으로 매칭하는 연출을 수행한다.

```
[Stage 1] 번개 연출
    - 같은 아이템 쌍을 잇는 번개/전기 이펙트
    - 복수 쌍이면 순차적으로 번개 연출 (각 쌍 0.5초)

[Stage 2] 강조 회전
    - 매칭된 쌍 카드가 글로우(발광) + 회전 애니메이션
    - 필러 카드는 페이드 아웃 또는 축소
    - 쌍 아이템에 집중하도록 시각적 강조

[Stage 3] 결과 확정
    - 쌍 아이템이 중앙으로 모여 최종 보상 표시
    - 가격대별 이펙트(8.5)와 함께 획득 확정 연출
    - 사운드: 가격대에 따른 차등 효과음
```

> **꽝(10장 전부 필러) 시**: 짝 맞추기 연출 스킵 → 꽝 전용 연출 → 결과 화면(8.9). (**Q1 재확정**: 꽝화면참고스샷.png 기반, 2026-03-16)
>
> **꽝 연출 상세** (레퍼런스: `꽝화면참고스샷.png`):
> 1. 카드 전부 공개 후 1초 대기
> 2. 중앙 제단에서 흰색 연기/폭발 파티클 분출 (Smoke + 번개 Spark, 약 1.5초)
> 3. 연기 위에 **"다음 기회에"** 대형 텍스트 팝업 등장
> 4. 하단 버튼: "닫기" 단일 버튼 → 뽑기 상점 복귀
> 5. 효과음: 아쉬운 톤의 짧은 효과음 (기존 기본 효과음 활용)

### 8.5 짝 맞추기 이펙트 색상 — 별(성) 등급별

짝 맞추기 연출(번개 + 글로우)의 이펙트 색상은 **장비 별(성) 등급** 기준으로 차별화한다. (**Q2 최종 확정**: v0.2 기획서 기준 — 별(성) 등급별 RGB, 2026-03-17)

> **가격대별 상자/개봉 연출 차등(8.6)은 유지**한다. 가격대 기준은 상자 외형과 개봉 연출에만 적용되고, 짝 맞추기 이펙트(번개/글로우) 색상은 별(성) 등급 기준이다.

#### 별(성) 등급별 이펙트 색상 (v0.2 확정)

| 등급 | RGB | 색상 | 번개 이펙트 | 글로우 이펙트 |
|:----:|-----|:----:|:----------:|:------------:|
| 1~3성 | 255,255,255 | 흰색 | 흰색 번개 | 흰색 글로우 |
| 4~6성 | 146,208,80 | 초록 | 초록 번개 | 초록 글로우 |
| 7~9성 | 0,176,240 | 파랑 | 파랑 번개 | 파랑 글로우 |
| 10~11성 | 204,0,0 | 빨강 | 빨강 번개 | 빨강 글로우 |

- **꽝(필러) 연출** (Q1 재확정, 꽝화면참고스샷.png 기반): 연기/폭발 파티클 + "다음 기회에" 대형 텍스트 + "닫기" 버튼
- **짝 맞추기 강조** (Q3 확정): 쌍 카드만 글로우(등급 색상) + 진동 연출, 필러는 원래 상태 유지
- **스페셜 상자** (Q6 확정): 일반/스페셜 이분 구분 — 스페셜 상자 파티클 3배 강도 (영상 0:23~0:29 참조)

### 8.6 가격대별 연출 차별화

| GachaPriceCode | 상자 외형 | 개봉 연출 | 배경 |
|:--------------:|:---------:|:---------:|:----:|
| 100 | 나무 상자 | 기본 개봉, 빠른 전개 | 기본 배경 |
| 1000 | 은 상자 | 중간 연출, 빛줄기 추가 | 중간 배경 이펙트 |
| 10000 | 금 상자 | 고급 연출, 전용 애니메이션 + 카메라 줌 | 전용 배경 + 파티클 |

### 8.6.1 소프트 피티 상자 연출 (스페셜 상자)

소프트 피티 발동 시 일반 상자 대신 **스페셜 상자**를 표시하여 고등급 결과임을 선제 예고한다. 연출 방식은 **이분(Binary)** — 일반 vs 스페셜 두 가지만 존재하며, 단계적 점진 연출은 없다. (**Q6 확정**: 영상 레퍼런스 기준, 2026-03-16)

| 조건 | 상자 유형 | 연출 |
|------|:--------:|------|
| 일반 뽑기 (피티 미발동) | 일반 상자 | 가격대별 연출 (8.6 기준) |
| 소프트 피티 발동 (연속 꽝 10회) | **스페셜 상자** | 특수 파티클(스노우플레이크 패턴 등) + 강화 사운드 + 카메라 줌 |

- 스페셜 상자 연출은 **상자 등장 단계(8.2 Step 1~2)** 에서 적용 (카드 배치 전)
- 유저에게 "이번 뽑기는 특별하다"는 기대감 선제 제공
- 실제 보상 확정은 카드 뒤집기 → 짝 맞추기 연출(8.4) 이후

## 9. 에러 처리

| 케이스 | 설명 | 처리 | 비용 |
|--------|------|------|:----:|
| 유효하지 않은 GachaPriceCode | 100, 1000, 10000 외 | `GameDatascriptDataIsNotExist` | 미차감 |
| 분해 불가능한 패턴값 | 유효 조합으로 분해 불가 | `GachaSystemError` + 로그 | 미차감 |
| Row 할당 실패 | 모든 PageGroup의 row 전부 할당 | 에러 반환 + 운영 알림 | 미차감 |
| 아이템 풀 부족 | RequiredClass 매칭 아이템 < 10개 | `GachaSystemError` + 로그 | 미차감 |
| 소프트 피티 발동 시 양수 슬롯 없음 | 남은 슬롯이 전부 0 | 꽝 처리 (피티 불발) + 로그 | 정상 차감 |

**원칙**: 에러 발생 시 **비용 미차감 (롤백)** + 운영 로그. 서버 기동 시 전수 검증으로 대부분의 에러를 사전 차단한다.

## 10. 가격대별 기대값 설계 가이드

### 10.1 권장 패턴 분포

| GachaPriceCode | 꽝 비율 | 당첨 평균값 | 최대 당첨 | 기대값(EV) | 하우스 엣지 |
|:--------------:|:-------:|:----------:|:---------:|:----------:|:----------:|
| 100 | 40~55% | 500~1000 | 2500~5000 | 60~80원 | 20~40% |
| 1000 | 30~45% | 1000~5000 | 25000~50000 | 700~850원 | 15~30% |
| 10000 | 15~25% | 10000~50000 | 500000~2500000 | 8000~9000원 | 10~20% |

### 10.2 역진적 하우스 엣지 구조

```
100원(20~40%) > 1000원(15~30%) > 10000원(10~20%)
```

저가 가챠는 높은 마진으로 운영 수익을 확보하고, 고가 가챠는 낮은 마진으로 고래 유저의 충성도를 유지한다.

### 10.3 가격대 간 전환 유도

- **100→1000**: 1000원 가챠에 100원에서 접근 불가한 아이템 등급(GachaItemPriceCode 5000+) 배치
- **1000→10000**: 10000원 전용 최상위 아이템(GachaItemPriceCode 500000~2500000) 배치

## 11. 규제 대응 전략

### 11.1 확률 표시 방법

패턴 기반이므로 전통적 "확률" 표시가 어렵다. **전체 평균 확률 공개** 방식을 채택한다.

표시 예시:
```
[100원 뽑기] 당첨 확률: 약 50% (전체 슬롯 기준)
  - 500원 상당: 30%
  - 1000원 상당: 15%
  - 2500원 이상: 5%
* 실제 결과는 배치에 따라 달라질 수 있습니다.
```

### 11.2 Row 간 분포 균일화

규제 대응을 위해 모든 row의 Pattern 분포가 **일정 범위 내에서 동일 구조**를 갖도록 설계한다:
- 꽝 비율 ±5% 이내
- 최대 당첨 1개 이상 보장
- 기대값 ±10% 이내

### 11.3 국가별 대응

| 국가 | 규제 | 대응 |
|------|------|------|
| 한국 | 게임산업법 확률 표시 의무 | 전체 평균 확률 공개 |
| 일본 | JOGA 자율규제 | 동일 방식 + 컴플리트 가챠 해당 없음 확인 |
| 유럽 (벨기에) | 유료 루트박스 금지 | 해당 국가 유료 가챠 비활성화 |
