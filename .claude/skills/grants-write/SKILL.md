# Grants Write — 정부과제 문서 작성 오케스트레이터

에이전트 팀을 편성하여 정부과제 문서를 작성한다. Lead(오케스트레이터)가 전체 파이프라인을 관리하고, 각 에이전트가 역할별로 작업을 수행한다.

## 에이전트 팀 구성

| 에이전트 | 파일 | 역할 | 모델 |
|---------|------|------|:---:|
| **Lead** (본 스킬) | - | 과제 분류, 팀 편성, 단계 관리, Human 소통 | opus |
| **Analyst** (분석관) | `~/.claude/agents/grants-analyst.md` | 원본 자료 전수 분석 + 보고서 + RAG 빌드 | sonnet |
| **Strategist** (전략관) | `~/.claude/agents/grants-strategist.md` | 과제 유형 판별 + 방향성 + 체계 프레이밍 | opus |
| **Writer** (작성관) | `~/.claude/agents/grants-writer.md` | 방향성 기반 본문 작성 + 근거 매핑 | sonnet |

## 파이프라인

### Phase 0. 과제 분류 + 환경 셋업 (Lead)

1. INDEX.md가 있으면 읽고 기존 컨텍스트 파악
2. `_grant-info.md` 읽기 → 과제 기본 정보 확인
3. 공고문 분석 → **과제 유형 자동 판별**:
   - R&D 기획형 / R&D 고도화형 / 사업화형 / 콘텐츠 제작형 / 기타
4. 판별 불확실 시 → Human에게 확인
5. 과제 유형을 모든 에이전트에 전달

### Phase 1. 자료 분석 (Analyst 스폰)

Analyst에게 전달:
- `_source/` 폴더 경로
- 과제 유형 (분석 관점 결정)
- **참고자료 경로**: `part2-목차.md` 상단 `[참고자료]` 섹션에 정의된 파일 목록을 Read하여 그대로 전달

Analyst 산출물:
- 폴더별 정밀분석 보고서
- SEMANTIC-INDEX.md
- RAG 인덱스

**체크포인트**: Analyst 완료 후 Lead가 보고서 검토. 누락된 자료가 있으면 재분석 지시.

### Phase 2. 방향성 수립 (Strategist 스폰)

Strategist에게 전달:
- 과제 유형
- Analyst의 정밀분석 보고서
- 공고문 + 사업설명회 자료 경로
- 검수자 방향성 문서 (있으면)

Strategist 산출물:
- 과제 프로필 (유형, 평가 기준, 작성요령)
- 방향성 문서 (기술/사업 체계 프레이밍)
- 개요 (part2-개요.md)

**[STOP] Human 확인**: 방향성과 체계를 Human에게 제시하고 승인을 받는다. 수정 피드백이 있으면 Strategist에게 재작업 지시.

### Phase 2.5. Writer 합격 기준 명시 (Rubric 선행)

> Generator(Writer)가 시작하기 전에 평가 기준을 명시한다. 기준이 모호하면 결과물도 모호해진다.

Writer가 달성해야 할 합격 기준 (grants-review 5축 기반):

| 축 | 배점 | 합격 기준 |
|---|:---:|---------|
| 축1: 작성요령 정합성 | 25% | **5단계 검증 통과** — L0(필수 정량 목표) + L1(대섹션 평가) + L2(세부 작성요령) + L3(교차) + L4(전체) |
| 축2: 데이터 근거 | 25% | 모든 수치·주장에 출처 명시 |
| 축3: 평가위원 시뮬레이션 | 25% | **4인 패널**(기술/시장·투자/정책/실행역량) 모든 관점에서 설득력 확보 |
| 축4: 톤·수위 | 15% | 과장 없음, 격식체 유지 |
| 축5: 검수자 방향성 | 10% | 검수자 피드백 파일 반영 |

**축1 3계층 검증 기준:**
- **L1**: 각 장 `[평가]` 태그의 상위 질문에 세부항목들이 답하고 있는지
- **L2**: 각 세부항목 `[작성요령]` 요구사항 1:1 반영
- **L3**: 대섹션↔세부항목 갭 없음, 1→4장 스토리라인 연결, 장 간 데이터 정합

**불합격 즉시 조건**: 작성요령 `<span>` 태그 삭제, 근거 없는 수치 생성, 방향성 문서 무시, **대섹션 [평가] 기준 미충족**, **과제 필수 정량 목표 누락/변경**

이 기준을 Writer에게 함께 전달한다.

### Phase 3. 본문 작성 (Writer 스폰)

Writer에게 전달:
- Strategist의 방향성 문서 + 개요
- **참고자료**: `part2-목차.md` 상단 `[참고자료]` 섹션을 Read하여 목록 그대로 전달
- 작성요령 (삭제 금지 규칙)
- **Phase 2.5 합격 기준 (Rubric)**

Writer 산출물:
- part2-본문.md
- 시각 요소 (개념도, 테이블)

### Phase 4. 변환 + 마무리 (Lead)

1. md → docx 변환 (md-to-docx.py)
2. docx 정합성 검수 (색상/헤딩/테이블/이미지)
3. 1페이지 요약 생성
4. INDEX.md 최종 업데이트
5. 검수자 최종본 대비 미반영 항목 체크 (있으면)

## 모든 자료 읽기 규칙

"자료"로 제공된 것은 형식 불문 **전부** 읽는다:
- .txt, .md, .docx, .hwp, .pdf, .pptx, .ppt, .xlsx, .xls, .csv, .html, .png, .jpg
- 문서 내 이미지/도형/차트/다이어그램도 추출하여 분석
- 못 읽는 형식 → Human에게 변환 요청
- **읽지 않은 자료가 있으면 안 된다**

## 과제 유형별 프레이밍

| 과제 유형 | Strategist 프레이밍 | Writer 톤 |
|---------|-----------------|---------|
| R&D 기획 | 기술 모듈 + KPI + PoC/PoV 가설 | 기획 수준, 방향성 |
| R&D 고도화 | 구현 계획 + 성과 지표 + 실증 | 구체적, 실행 계획 |
| 사업화 | BM + 매출 + 투자 + GTM | 사업 전략, 성과 중심 |
| 콘텐츠 제작 | 제작 계획 + 유통 + 수익 | 창작 역량, 시장성 |

## Phase F. 피드백 반영 루프

Human 또는 검수자로부터 피드백이 오면 Lead가 분류하여 적절한 에이전트에 라우팅한다.

### 피드백 분류 + 라우팅

| 피드백 유형 | 라우팅 | 예시 |
|-----------|------|------|
| **방향성 변경** | Strategist 재프레이밍 → Writer 재작성 | "3단계를 1,000종 전수로", "기술 체계 바꿔" |
| **기술 보강** | Writer + /rag-search로 근거 검색 | "7대 모듈 상세 추가", "검수자 문서에서 가져와" |
| **톤/표현** | Writer 직접 수정 | "SaaS 빼", "과장 표현 수정", "기획 수준 톤으로" |
| **단순 수정** | Lead 직접 반영 | 오탈자, 수치 수정, 문단 순서 |
| **자료 추가** | Analyst 추가 분석 → RAG 업데이트 → Writer 반영 | "새 파일 추가됐어", "이 문서도 읽어" |

### 피드백 기록

모든 피드백은 반드시 기록한다:
- `_source/08_참고자료/검수자-채팅피드백-{날짜}.md` — 피드백 원문 누적 (없으면 새 파일 생성)

### 피드백 반영 후 처리

1. 본문 수정 완료
2. 검수자 최종본 docx에서 새로 사용한 항목을 빨간색으로 표시
3. INDEX.md 업데이트

## INDEX.md 관리

- 세션 시작 시 INDEX.md 먼저 Read
- 각 Phase 완료 시 INDEX.md 업데이트
- INDEX.md가 없으면 생성하지 않고 경고만 출력
