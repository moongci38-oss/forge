# Grants Review — 정부지원사업 문서 검수

작성된 연구개발계획서를 5개 축으로 병렬 검수하고, 통합 리포트를 출력한다. 3개 서브에이전트가 각각 담당 축을 독립적으로 검수하고, 결과를 통합하여 단일 리포트로 합친다.

## 입력

| 입력 | 필수 | 설명 |
|------|:---:|------|
| **본문** | O | 검수 대상 본문 파일 경로 |
| **분석 보고서** | △ | 근거 데이터 분석 보고서. **미지정 시 자동 탐색** (아래 참조) |
| **평가 기준** | △ | 배점·평가관점 (없으면 기본 R&D 기준 적용) |
| **검수자 방향성** | △ | 검수자 철학·키워드·금지어 |
| **이전 리포트** | △ | 이전 검수 리포트 (있으면 개선 diff 표시) |

## 참고자료

`part2-목차.md` 상단 `[참고자료]` 섹션을 Read하여 사용한다. 참고자료 목록은 목차에서만 관리한다.

`*-검수리포트-*.md` 패턴은 이전 리포트로 취급한다 (참고자료에서 제외).

## Evaluator 핵심 원칙: 절대 관대하게 보지 마라

아래 생각이 들면 그것은 관대해지고 있다는 신호 → 더 엄격하게 본다:
- "나쁘지 않은데..." → 감점
- "이 정도면 괜찮지 않나?" → 감점
- "전반적으로 잘 썼으니 이 부분은 넘어가자" → 금지

행동 규칙:
- 한 축이 좋아도 다른 축의 문제를 상쇄하지 않는다
- 첫인상이 좋아도 5축 항목을 반드시 하나씩 검증한다
- 작성자의 자체검토("작성요령 준수했음")를 그대로 믿지 않는다 — 직접 1:1 대조한다

### 피드백 형식: 위치 + 이유 + 방법 (3요소 필수)

모든 이슈는 반드시 세 요소를 포함한다:
- **나쁜 예**: "표현이 과장되었습니다"
- **좋은 예**: "2-2절 3번째 문단 '업계 최초' 표현 (위치) → 근거 데이터 없는 자화자찬 (이유) → '국내 최초 적용 사례로 〇〇특허 출원(2025) 기반' 으로 수정 (방법)"

수정 우선순위 TOP N의 각 항목은 이 3요소를 채워야 한다.

## 검수 5축

### 축 1: 작성요령 정합성 (5단계 검증)
작성요령을 **필수수치→상위→하위→교차→전체** 5단계로 검증한다:
- **Layer 0: 필수 정량 목표** — 공고문→사업설명회→검수자 피드백 순으로 필수 수치 수집, 본문 1:1 대조 (CRITICAL)
- **Layer 1: 대섹션 [평가]** — 각 장 시작부 `[평가]` 태그의 상위 목표 수집
- **Layer 2: 세부 [작성요령]** — 1-1~4-3 각 항목의 작성요령 1:1 대조 ("반드시"→필수, "지양"→위반)
- **Layer 3: 계층 교차** — 대섹션 평가 기준을 세부항목이 충분히 커버하는지, 항목 간 중복·모순
- **Layer 4: 문서 전체** — 1→4장 스토리라인 연결, 장 간 데이터 정합, 분량 균형

### 축 2: 데이터 근거 검수
분석 보고서의 데이터가 본문에 적절히 인용되었는지 교차 검증한다.
- 수치 팩트체크 (원본 대조)
- 출처 불명 수치 탐지
- 미활용 핵심 데이터 TOP 5 도출

### 축 3: 평가위원 시뮬레이션 (4인 패널 다관점)
**4개 관점**(기술/시장·투자/정책/실행역량)에서 독립 채점한다:
- 관점별 항목별 예상 점수 + 강점/약점/개선 제안
- **관점 편차 경고**: 특정 관점에서만 낮은 항목 = 맹점 → 우선 개선
- 최종 점수 = 4관점 평균, 서면·발표·최종 선정 가능성 판단
- 핵심 개선 우선순위 (점수 영향도 + 관점 표시)

### 축 4: 톤·수위 검수
기획 단계에 맞는 톤인지 점검한다.
- 과장 표현 ("혁신적", "최초의", "초고속")
- 단정적 표현 (확정 계획 톤 vs 방향성 수준)
- 자화자찬 표현 (근거 없는 자기 평가)

### 축 5: 검수자 방향성 반영
검수자의 핵심 철학·키워드가 본문에 녹아들었는지 확인한다.
- 철학별 반영도 (%)
- 키워드 포함 여부 체크리스트
- 금지어 사용 여부

**검수자 피드백 파일 체크**: 본문과 같은 디렉토리 또는 `_source/08_참고자료/`에서 `검수자-채팅피드백*.md` 파일을 탐색한다. 존재하면 파일 내의 개별 결정사항을 항목별로 1:1 대조하여 본문 반영 여부를 체크한다. 특히:
- 검수자가 명시적으로 결정한 사항 (예: "PoV는 자체 IP로", "SaaS 금지")
- 검수자가 "논의 필요"로 남긴 미확정 사항 — 본문에서 확정했다면 그 사실을 리포트에 명시
- 검수자의 우려 사항 (예: "상용으로 볼 수 있다") — 본문 표현이 우려를 회피했는지 체크

## 실행 방식

3개 서브에이전트를 **병렬 스폰**한다:

| 에이전트 | 파일 | 담당 축 |
|---------|------|--------|
| `grants-review-guidelines` | `~/.claude/agents/grants-review-guidelines.md` | 축 1 + 축 4 |
| `grants-review-data` | `~/.claude/agents/grants-review-data.md` | 축 2 + 축 5 |
| `grants-review-evaluator` | `~/.claude/agents/grants-review-evaluator.md` | 축 3 |

### 스폰 시 전달할 정보

**Agent A (guidelines):**
- DOCUMENT_PATH: 본문 경로

**Agent B (data):**
- DOCUMENT_PATH: 본문 경로
- REFERENCE_PATHS: P1 참고자료 경로들 (검수자 피드백·최신본·선행자료)
- REVIEWER_DIRECTION: 검수자 방향성 (있으면)

**Agent C (evaluator):**
- DOCUMENT_PATH: 본문 경로
- EVAL_CRITERIA: 평가 기준 (있으면)

### 통합 방식

3개 에이전트 결과를 수신한 후:

1. 축 1~5 결과를 순서대로 병합
2. 총점 계산 (축 3의 예상 점수 기준)
3. **수정 우선순위 TOP N** 생성 — 모든 축의 이슈를 점수 영향도 순으로 정렬
4. 이전 리포트가 있으면 **개선/미개선 diff** 추가

## 리포트 저장

검수 결과는 본문과 같은 디렉토리에 저장한다:
`{본문파일명}-검수리포트-{YYYY-MM-DD}.md`

예: `part2-본문-검수리포트-2026-03-29.md`

## INDEX.md 자동 관리

검수 완료 후, 해당 과제의 `INDEX.md`를 업데이트한다:

1. INDEX.md 위치: `{과제 루트}/INDEX.md`
2. 업데이트 항목:
   - 최신 검수 리포트 파일명 + 점수
   - 프로젝트 상태 (현재 단계, 본문 점수, 최종 수정일)
3. INDEX.md가 없으면 생성하지 않는다 — 경고만 출력

## 세션 시작 규칙

과제 작업 시작 시 INDEX.md를 먼저 Read하여 컨텍스트를 파악한다.
INDEX.md가 있으면 파일 탐색(find/ls) 없이 바로 작업 진입 가능.
