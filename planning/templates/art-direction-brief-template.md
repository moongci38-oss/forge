## 7. 참조 이미지 사용 프로토콜

### 앵커 이미지 vs 생성 이미지 분리

| 구분 | 앵커 이미지 (Reference) | 생성 이미지 (Generated) |
|------|----------------------|----------------------|
| 역할 | 스타일 기준점 | 신규 에셋 |
| 출처 | 기존 승인 에셋 (§3 앵커 이미지) | MCP 도구 생성 |
| 사용법 | Gemini 프롬프트에 참조로 첨부 | 프로젝트 에셋으로 배치 |
| 수정 | 절대 수정 금지 | 피드백 반영 수정 가능 |

### 참조 이미지 활용 흐름

```
1. 에셋 생성 요청
2. 가장 유사한 앵커 이미지 선택 (§3 앵커 이미지 참조)
3. Gemini 프롬프트에 앵커 이미지 첨부:
   "Generate [에셋 설명], style matching the attached reference image.
    Maintain the same [색상 톤/질감/조명 방향]."
4. 생성 결과를 앵커와 비교하여 일관성 검증
```

### 참조 이미지 업데이트 주기

- P2 프로토타입 승인 후 → 초기 앵커 3장 선정
- P3 매 5장 승인 시 → 앵커 후보 리뷰 (더 대표적인 에셋으로 교체 가능)
- P4 검증 완료 시 → 최종 앵커 확정

## 8. 의도적 긴장 규칙 (Intentional Tension)

> "완벽한 균형은 지루하다. 영혼이 있는 디자인은 한 곳이 의도적으로 어긋나 있다."

| 긴장 기법 | 설명 | 프롬프트 키워드 예시 |
|----------|------|-------------------|
| **색상 악센트 이탈** | 팔레트 95% 준수, 5% 예상 밖 컬러 | `unexpected warm {color} accent amid {dominant_palette}` |
| **구도 비대칭** | 핵심 요소를 정중앙에서 살짝 이탈 | `positioned slightly off-center, rule of thirds` |
| **스케일 의외성** | 하나의 요소만 비정상적으로 크거나 작게 | `dramatically oversized {element}, tiny {contrast_element}` |
| **빛의 방향 교란** | 주 광원 외 예상 밖 보조 광원 | `unexpected {light_source} from {direction}` |
| **질감 대비** | 매끈한 것과 거친 것의 병치 | `smooth {material_a} against rough {material_b}` |
| **빈 공간의 힘** | 채우지 않는 것도 디자인 결정 | `vast empty {space} around {subject}, isolation amplifying presence` |

> 프로젝트별로 위 6가지 중 2-3가지 핵심 긴장 기법을 선택하여 일관되게 적용한다.

## 9. 외주 브리프 원칙 (Why & What, Not How)

> SDT 원칙: 외주 파트너의 자율성을 보존하면 결과물 품질이 높아진다.
> 이 브리프는 **왜**와 **무엇을**만 정의한다. **어떻게**는 파트너가 결정한다.

| 항목 | 내용 | 작성 원칙 |
|------|------|----------|
| **Why** (목적) | {이 에셋이 왜 필요한가 — 비즈니스 맥락, 사용 시나리오} | 1-2문장. "~때문에 필요" |
| **What** (완성 기준) | {deliverable 형태, 검수 기준, 파일 스펙} | 측정 가능한 기준만. 추측 여지 제거 |
| **How (작성 금지)** | — | 제작 방식·도구·프로세스는 파트너 자율 선택 |

> **체크**: "이 브리프에 제작 방법을 지시하는 문장이 있는가?" → 있으면 삭제.
> 파트너가 방법을 선택할 때 공헌감을 느끼고 더 나은 결과를 낸다.
