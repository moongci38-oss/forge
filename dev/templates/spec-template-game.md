## 4. 플레이어 플로우

> [작성 가이드] 각 주요 기능에 대해 Happy Path, Error Path, Edge Case를 분리하여 작성한다. 플로우 단계는 번호를 매기고, 각 단계에서 플레이어 액션과 시스템 반응을 명시한다.

### 4.1 [플로우명 1]

#### Happy Path (정상 흐름)

> [작성 가이드] 플레이어가 목표를 달성하는 가장 일반적인 경로를 단계별로 기술한다.

```
1. 플레이어: [액션]
   → 시스템: [반응]
2. 플레이어: [액션]
   → 시스템: [반응]
3. 완료 → [최종 결과]
```

#### Error Path (에러 흐름)

> [작성 가이드] 예상되는 에러 상황과 시스템의 에러 처리 방식을 기술한다. 플레이어에게 보이는 에러 메시지와 복구 경로를 포함한다.

| 에러 상황 | 발생 시점 | 시스템 반응 | 플레이어 복구 경로 |
|----------|----------|-----------|-----------------|
| [예: 네트워크 끊김] | [단계 N] | [재연결 시도 + 안내 팝업] | [자동 재연결 / 타이틀로 복귀] |
| [예: 재화 부족] | [단계 N] | [부족 알림 + 상점 안내] | [상점 이동 / 취소] |
| [예: 서버 타임아웃] | [단계 N] | [재시도 안내] | [재시도 / 메인 복귀] |

#### Edge Case (경계 조건)

> [작성 가이드] 정상도 에러도 아닌 경계 상황을 기술한다. 동시 입력, 빈 데이터, 극단값 등.

| 경계 상황 | 기대 동작 |
|----------|----------|
| [예: 연출 중 뒤로가기] | [연출 스킵 후 결과 표시] |
| [예: 동시에 같은 아이템 사용] | [첫 요청만 처리, 중복 차단] |
| [예: 재화 최대치 초과 획득] | [최대치까지만 적용 + 알림] |

## 5. 데이터 모델

> C# 클래스/구조체로 정의한다. 서버-클라이언트 공유 모델은 직렬화 방식을 명시한다.

> [작성 가이드] 프로젝트의 데이터 파이프라인 구조에 따라 레이어를 정의한다.
> 프로젝트마다 아키텍처가 다를 수 있으므로, 실제 사용하는 레이어만 기재한다.
>
> | 레이어 | 역할 | 네이밍 패턴 | 예시 |
> |--------|------|-----------|------|
> | [Data/Model] | 단일 레코드 모델 | [프로젝트 네이밍 규칙] | [예: ShopGachaData.cs] |
> | [DAO/Repository] | 직렬화/역직렬화, DB 접근 | [프로젝트 네이밍 규칙] | [예: ShopGachaDao.cs] |
> | [Table/Collection] | 여러 레코드 배열 | [프로젝트 네이밍 규칙] | [예: ShopGachaTable.cs] |
> | [Store/Cache] | 런타임 캐시, 접근 인터페이스 | [프로젝트 네이밍 규칙] | [예: DataManager.ShopGacha] |
>
> **직렬화 필수 체크리스트**:
> - [ ] 서버-클라이언트 공유 모델은 프로젝트 직렬화 방식 적용 (섹션 6 참조)
> - [ ] 왕복 테스트 (객체 → 바이트 → 객체) 일치 확인
> - [ ] 민감 데이터(확률, 재화) 서버 측 계산, 클라이언트는 표시만

> | 방식 | 사용 시점 | 예시 |

### 9.5 UI 상태

> [작성 가이드] Canvas 계층, UI 팝업, 에러/로딩 표시, 전환 애니메이션, 인터랙션 피드백을 정의한다.

#### 9.5.1 UI 전환 애니메이션

> [작성 가이드] 각 UI 요소의 진입/퇴장 트윈 파라미터를 구체적 수치로 정의한다. "부드럽게 나타남" 같은 추상적 서술 금지 (forge-spec-visual-binding.md C-1).

| UI 요소 | 전환 유형 | Property | From | To | Duration(s) | Ease | 비고 |
|--------|---------|----------|------|-----|:-----------:|------|------|
| [예: 메인 패널] | 진입 | Scale | 0.8 | 1.0 | 0.25 | EaseOutBack | CanvasGroup.alpha 동시 0→1 |
| [예: 메인 패널] | 퇴장 | Alpha | 1.0 | 0.0 | 0.15 | EaseInSine | interactable=false 선행 |
| [예: 결과 팝업] | 진입 | AnchorPosY | -Screen.height | 0 | 0.3 | EaseOutCubic | 하단에서 슬라이드 업 |

#### 9.5.2 UI 상태 다이어그램

> [작성 가이드] 주요 UI 패널/팝업의 상태 전이를 Mermaid stateDiagram으로 정의한다. 상태 수 3개+ 또는 복잡한 전환 조건이 있으면 Element Task Doc 분리를 검토한다.

```mermaid
stateDiagram-v2
    [*] --> Hidden
    Hidden --> Entering: Show 호출
    Entering --> Visible: 트윈 완료
    Visible --> Exiting: Close 호출
    Exiting --> Hidden: 트윈 완료
    Visible --> Refreshing: 데이터 갱신
    Refreshing --> Visible: 갱신 완료
```

| 상태 | 활성 오브젝트 | 입력 허용 | 전환 조건 |
|------|-------------|:--------:|----------|
| [예: Hidden] | 없음 | N | Show() 호출 |
| [예: Entering] | [패널 + 로딩] | N | 진입 트윈 완료 |
| [예: Visible] | [패널 + 콘텐츠] | Y | Close() 또는 데이터 갱신 |
| [예: Exiting] | [패널 fade out] | N | 퇴장 트윈 완료 |

### 9.9 연출/이펙트 스펙

> [작성 가이드] 파티클, 셰이더, 트윈 시퀀스, 사운드 트리거, 타임라인 등 연출 요소를 정의한다.
> [작성 가이드] 구현 가이드 컬럼에 프로젝트에서 사용하는 트윈 라이브러리를 명시한다.
> - Unity 신규 프로젝트: DOTween Sequence 권장
> - NGUI 기존 프로젝트: UITweener 유지 or DOTween 병용 (프로젝트 정책에 따라)
> - 트윈 라이브러리 혼용 금지 (프로젝트 내 하나로 통일)
>
> Complex 연출(타임라인 4스텝+, 등급 분기, 파티클+사운드 동기화)은 Element Task Doc으로 분리한다 (9.9.9 참조).
> 구현 패턴 참조: `forge/shared/docs/forge-direction-ui-implementation-guide.md`

#### 9.9.1 연출 타임라인

> [작성 가이드] 연출의 시간 흐름을 시퀀스 번호로 정의한다. 이 테이블이 DOTween Sequence의 Append/Join/Insert와 1:1 매핑되어야 한다.

| 시퀀스 # | 시작(s) | 종료(s) | 대상 | 액션 | 파라미터 | 관계 |
|:--------:|:-------:|:-------:|------|------|---------|:----:|
| [예: 1] | 0.0 | 0.5 | [상자] | [Scale 등장] | [0→1.2, EaseOutBack] | — |
| [예: 2] | 0.5 | 0.65 | [상자] | [Scale 안정] | [1.2→1.0, EaseInSine] | After #1 |
| [예: 3] | 0.65 | — | [상자] | [탭 대기] | [Tap 입력 대기] | After #2 |
| [예: 4] | — | — | [카드] | [Y축 회전] | [0→180°, 0.6s, EaseInOutSine] | After #3 (탭 시) |

#### 9.9.2 트윈 파라미터 상세

> [작성 가이드] 추상적 서술 금지 (forge-spec-visual-binding.md C-1). "부드럽게" → "EaseInOutSine, 0.3s". 모든 트윈을 빠짐없이 기재한다.

| 대상 | Property | From | To | Duration(s) | Ease | Delay(s) | Loop | 비고 |
|------|----------|------|-----|:-----------:|------|:--------:|------|------|
| [예: 상자] | Scale | 0 | 1.2 | 0.5 | EaseOutBack | 0 | — | 등장 |
| [예: 상자] | Scale | 1.2 | 1.0 | 0.15 | EaseInSine | 0 | — | 안정 |
| [예: 카드] | RotationY | 0 | 90 | 0.3 | EaseInSine | 0 | — | 플립 전반 |

## 10. 테스트 요구사항

> [작성 가이드] 각 테스트 레벨에서 무엇을 검증하는지, 구체적 시나리오 예시를 포함하여 작성한다.

### 10.1 EditMode 테스트

> [작성 가이드] MonoBehaviour 없이 순수 로직을 검증한다: 알고리즘, 상태 전이, 데이터 파싱, 확률 계산 등.

#### 핵심 로직 테스트

| 테스트 대상 | 시나리오 | 기대 결과 |
|-----------|---------|----------|
| [예: 분해 알고리즘] | 정상 입력 (아이템 5개, 등급 R) | 분해 재화 N개 반환 |
| [예: 확률 테이블 파서] | 정상 기획 데이터 | 확률 합계 100% |
| [예: 직렬화 왕복] | 객체 → 바이트 → 객체 왕복 | 원본과 동일 |

| 테스트 대상 | 경계 조건 | 기대 결과 |
|-----------|----------|----------|

| [예: 확률 계산] | 보정 확률 최대치 | 100% 초과 방지 |
| [예: 인벤토리] | 최대 슬롯 초과 | 추가 차단 + 알림 |
